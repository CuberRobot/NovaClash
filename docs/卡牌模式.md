# NovaClash 卡牌模式设计

本文档描述「卡牌模式」的目标、与现有流程的关系，以及实现拆解。卡牌模式类似炉石传说：用卡牌可视化角色与战斗过程，在卡面上展示 HP、ATK、状态等，选角与增益通过图形化点击/拖拽卡面完成。

| 竖版卡面视觉参考 | [生成游戏角色竖版卡牌.png](./img/生成游戏角色竖版卡牌.png)、[生成游戏角色竖版卡牌 (1).png](./img/生成游戏角色竖版卡牌%20(1).png) | 竖版卡牌三区结构：顶部条带（金边/星标）、中部角色插图、底部信息区（HP/ATK/状态）；风格统一、便于实现与扩展。 |

## 一、目标与范围

### 1.1 核心目标

- **战斗可视化**：回合结算后，用**战场上的卡牌**展示双方 3v3 单位，卡面实时显示每个角色的 HP、ATK、标签与状态（中毒、护盾次数、存活/阵亡等）。
- **选角与增益图形化**：选角阶段用**卡牌**表示角色池与出击位，通过点击/拖拽卡牌完成「选 3 人 + 排顺序」；增益阶段通过点击卡面为对应出击位分配 ATK+2 或 HP+4，剩余次数与已分配列表图形化展示。
- **事件驱动卡面更新**：后端下发的 `round_result.events` 与现有事件类型不变，前端根据事件序列逐条更新卡牌状态（扣血、死亡、中毒、护盾、复活等），并可配合简单动画（伤害数字、高亮目标、阵亡效果）增强可读性。

### 1.2 与现有实现的关系

- **协议与后端**：不改变 WebSocket 消息格式与后端逻辑。仍使用 `round_start`（pool）、`submit_round`（selection、gains、strategy）、`round_result`（events、texts、winner、score）。
- **入口方式**：可以是「同一页面内切换视图」（当前列表/表单 + 卡牌视图切换），或独立路由/页面（如 `/static/card.html`），由项目偏好决定；文档按「独立卡牌视图」拆解，便于与现有 `index.html` 解耦。
- **数据来源**：卡牌模式仍消费 `state.pool`、`round_result.events` 等，仅展示层从「列表+文字战报」改为「卡牌+战场」。

## 二、卡牌与战场抽象

### 2.1 单张卡牌数据与展示

一张「角色卡」在任意阶段可统一用以下数据描述（与后端角色 dict 对齐）：

| 字段       | 说明                     | 来源 |
|------------|--------------------------|------|
| id         | 角色 id                  | pool/ROLES |
| name       | 名称                     | pool |
| atk        | 当前攻击力               | pool → 战斗中被增益/不变 |
| hp / max_hp| 当前血量 / 最大血量       | pool → 战斗中变化 |
| tags       | 标签 id 列表             | pool，战斗开始后可能被剥夺 |
| 状态扩展   | 中毒剩余轮次、毒伤害、护盾剩余次数、是否存活 | 由 events 推导或本地状态 |

**卡面展示建议（最小可行）**：

- **竖版卡牌**：卡牌为**竖版**（portrait 比例，高大于宽），与 [卡面参考图](./img/生成游戏角色竖版卡牌.png) 及 [战斗效果 demo](./img/card-mode-effect.png) 一致。
- **角色外观插图**：每张角色卡须有该角色的**外观插图/立绘**，作为卡面主体视觉（参考上述卡面参考图的中部插图区）；与设定集（星陨、星灵战士等）风格统一，按角色 id 绑定立绘资源（见 [§六、技术实现形式](#六技术实现形式)）。
- **三区结构**：卡面自上而下分为**顶部条带**（标识/装饰）、**中部插图区**（立绘）、**底部信息区**（HP、ATK、标签、状态），与参考图一致，便于实现与扩展。
- 底部信息区：角色名、ATK / HP（如 `4 / 22`），HP 可用进度条或颜色区分当前/最大；标签用简短文字或图标（护盾、中毒、狂暴等）；状态如中毒图标+轮次、护盾剩余次数；阵亡时整卡置灰或翻转、半透明，并标注「阵亡」。

扩展时可为每张卡增加稀有度边框、动态光效等。

### 2.2 两种布局

- **选角/增益阶段（Setup）**  
  - **角色池**：多张卡牌（当前 6 张）横向或网格排列，可点击选中/取消，选中态高亮；再通过「出击位区域」的 3 个卡槽确定顺序（点击池中卡放入槽位，或拖拽放入）。  
  - **出击位**：3 个槽位，每个槽位一张卡（或占位符）。槽位内卡牌可显示即将生效的增益（如 +2 ATK、+4 HP 图标），并支持点击槽位卡牌分配增益（先选增益类型，再点卡面）。  
  - 与现有逻辑对应：`state.selected`（1-based 池下标）× 顺序 = 出击位 1～3；`state.buffs` 为 `[出击位, 类型]`，共 4 次。

- **战斗展示阶段（Battlefield）**  
  - 双方各 3 张卡：我方 3 张一排，对方 3 张一排（可上下或左右对称），位置与出击位 1～3 对应。  
  - **己方视角排布**：从己方视角看，**己方角色按出击顺序从左到右**排列（第 1 位在最左，第 3 位在最右）；**对方角色按出击顺序从右到左**排列（对方第 1 位在己方视野的右侧，对方第 3 位在己方视野的左侧），使双方「相对而立」、便于理解对位关系。  
  - 根据 `round_result.events` 在本地维护「双方 3 个单位的当前 atk、hp、tags、中毒、护盾、存活」等状态，每收到一条事件就更新对应卡牌，并配合动画特效（见下）。

## 三、事件与卡牌状态同步

后端事件类型已固定，卡牌模式需要做的是：**从 events 还原或维护双方 6 个单位的实时状态**，并驱动卡面更新。

### 3.1 事件与状态变更对照（简要）

| 事件类型         | 含义           | 对卡牌状态的影响 |
|------------------|----------------|------------------|
| FIRST_STRIKE     | 先手方         | 仅 UI 高亮/标识先手方，不改卡数据 |
| CURSE_REMOVE_TAG | 剥夺目标标签   | 目标角色卡 tags 清空 |
| ROUND            | 新一轮次       | 仅 UI 轮次标题；可选中毒结算已在 POISON_TICK 体现 |
| POISON_TICK      | 中毒扣血       | 目标 hp 减少，若致死则触发 DIE |
| POISON_APPLY     | 施加中毒       | 目标中毒轮次与伤害更新 |
| SKIP_DEAD        | 单位已死跳过   | 目标存活=false（若尚未被 DIE 更新） |
| SELF_DESTRUCT    | 自爆           | 攻击方存活=false |
| SHIELD_TRANSFER  | 护盾分担       | 护盾者 shield_charges-1；主目标与护盾者 hp 更新由 HIT/后续 DIE 体现 |
| ARMOR_REDUCE     | 重装减伤       | 仅文案；实际伤害在 HIT 中已是减伤后数值 |
| PIERCE           | 穿透追加伤害   | 第二目标 hp 减少（或由后续 HIT 体现） |
| AOE_HIT          | 群伤           | 每个目标一条，目标 hp 减少 |
| HIT              | 单次命中       | 目标 hp 减少 |
| DIE              | 阵亡           | 目标存活=false，hp=0 |
| REVIVE           | 复活           | 目标存活=true，hp 设为复活后值 |

实现方式：在收到 `round_result` 时，用 `events` 在前端跑一遍「战斗回放」：维护 6 个单位的当前状态，按事件顺序更新；每更新完一条（或一批）可刷新一次卡面，并可触发该条事件对应的小动画（见下）。这样无需后端改协议即可得到每张卡在任意时刻的数值。

### 3.2 初始状态

战斗开始前，双方队伍来自本回合提交的 selection + gains。己方 3 张卡由 `state.selected` + `state.buffs` 可计算；对方 3 张卡在回合结果前不可见，可在 `round_result` 时由服务端在 payload 中附带「对方队伍摘要」（当前若未提供，可在后续扩展中加 `opponent_team: [{name, atk, hp, tags}, ...]`），或仅用 events 反推：从 HIT/AOE_HIT 等事件中的 `def_team_index, def_name` 等还原对方 3 个单位的 name，再通过事件逐步推 hp/存活。若后端暂时不返对方队伍，首版可只显示「对方 3 张卡」的 name/初始 atk/hp 由事件反推的近似值，或仅显示名称与血量变化。

## 四、交互与动效（可选分级）

### 4.1 选角与增益

- **选角**：池中卡牌点击 → 切换选中；已选卡按顺序进入 3 个出击位卡槽；支持从槽位移除回池。与现有 `toggleSelect`、`state.selected` 一致，仅 UI 改为卡牌。  
- **增益**：选择「ATK+2」或「HP+4」后，点击出击位上的卡牌即为该位添加一次增益；已分配列表用图形展示（如每张卡角标 4 个小图标，亮起表示已分配），剩余次数 4→0 与提交校验与现有一致。

### 4.2 战斗回放与动画特效

- **自动步进**：收到 `round_result` 后，可按事件顺序自动播放（每 0.8～1.5 秒一条），或提供「下一步」「播放/暂停」按钮。  
- **卡牌过程动画特效（必做）**：战斗过程中卡牌需具备与事件对应的动画，增强可读性与沉浸感，例如：  
  - **发起攻击**：攻击方卡牌短暂前移/缩放、攻击光效或弹道指向目标，再归位。  
  - **受到攻击**：受击卡牌震动、闪红或边框高亮，飘字显示伤害数值（如 `-8`），HP 条或数值变化带过渡动画。  
  - **群体伤害（AOE）**：多个目标依次或同时播放受击反馈。  
  - **阵亡（DIE）**：卡牌置灰、翻转、缩小或淡出，并标注「阵亡」。  
  - **复活（REVIVE）**：卡牌从灰恢复、血量条填充动画。  
  - **中毒/护盾等**：施加或结算时对应卡牌上的状态图标出现/更新动画。  
- 实现上可用 CSS `transition`/`animation` 配合 JS 在事件步进时为对应卡牌添加 class 或 data 属性，必要时辅以简单动画库。

## 六、技术实现形式

以下结合 [卡面参考图](./img/生成游戏角色竖版卡牌.png)（及 [另一张](./img/生成游戏角色竖版卡牌%20(1).png)）与 [战斗效果 demo](./img/card-mode-effect.png) 给出可落地的技术形态与实现要点。

### 6.1 卡面结构（三区）

参考竖版卡牌图，单张卡在 DOM 与样式上分为三块，便于复用与状态绑定：

| 区域 | 用途 | 实现要点 |
|------|------|----------|
| **顶部条带** | 稀有度/先手标识、或预留成本/能量；视觉上为深色条+金色星标/加号等 | 固定高度条带，背景色与金边与参考图一致；可选 `data-first="1"` 标识先手单位 |
| **中部插图区** | 角色外观立绘，卡面主体 | 占满竖版中间区域，`background-image` 或 `<img>` 按 `role.id` 映射；宽高比固定（如 3∶4），避免拉伸 |
| **底部信息区** | HP、ATK、标签、状态（中毒/护盾/阵亡） | 浅色底或与角色主色协调；HP 可用进度条+数字，ATK 数字；标签用图标或短文案；选中态可在此区或整卡加发光（参考图中底部蓝光） |

- **竖版比例**：整卡容器宽高比约 2∶3 或 3∶4（高 > 宽），用 CSS `aspect-ratio` 或固定 `height` 控制，与参考图一致。  
- **状态样式**：通过为卡根节点添加 class 或 `data-state` 驱动，例如 `data-state="selected"`（选中）、`data-state="dead"`（阵亡）、`data-state="poisoned"`（中毒），由 CSS 控制边框高亮、置灰、半透明等。

### 6.2 立绘与资源映射

- **路径约定**：按角色 id 绑定立绘，例如 `static/card-art/{id}.png` 或 `static/card-art/{id}.webp`；与 `ROLES`/`pool[].id` 一致。  
- **首版**：可共用一张占位图或按类型（均衡/输出/前排等）使用少量通用图；后续再为 1～12（及扩展 13～20）配备独立立绘。  
- **参考图风格**：卡面参考图为低多边形/ stylized 幻想风格，与设定集「星陨」「星灵战士」契合；正式资源建议统一色调与构图（竖版全身/半身），以便与顶部条带、底部信息区风格统一。

### 6.3 战场与选角布局的 DOM 结构

- **选角阶段**：一个「角色池」容器（如 `.pool-cards`）内 6 张竖卡；一个「出击位」容器（如 `.slots`）内 3 个槽位，每槽可放入一张卡或占位符。卡牌由 JS 根据 `state.pool` 生成，点击写入 `state.selected` 并更新槽位展示。  
- **战斗阶段**：两个区域，如 `.battlefield-mine`（己方）、`.battlefield-opponent`（对方）；己方区域内 3 张卡按顺序从左到右排列，对方区域 3 张卡按顺序从右到左排列（CSS 可用 `flex-direction: row` + 对方容器 `flex-direction: row-reverse`，或子节点 order 控制）。  
- **数据绑定**：每张卡对应一个单位状态对象（含 atk、hp、max_hp、tags、poison、shield_charges、alive 等）；事件步进时更新该对象并调用一次「刷新该卡 DOM」或仅更新数值/进度条节点，避免整卡重绘。

### 6.4 动画与事件反馈

- **实现方式**：以 CSS 为主，JS 在事件步进时为对应卡牌添加/移除 class 或设置 `data-*`，触发过渡或关键帧动画。  
- **攻击/受击**：攻击方加 `.card-attacking`（前移+缩放+光效），受击方加 `.card-hit`（震动+闪红+飘字容器显示 `-N`）；动画结束（如 300～500ms）后移除 class。飘字可用绝对定位的 span，用 CSS animation 做上浮+淡出。  
- **阵亡/复活**：阵亡加 `.card-dead`（灰度滤镜、翻转或缩小）；复活时移除 `.card-dead` 并加 `.card-revive`（血量条填充动画）。  
- **可选**：若需更复杂时间轴（多目标依次动效、延迟顺序），可引入轻量动画库（如 animate.css + 少量 JS），或使用 Web Animations API。

### 6.5 技术选型与工程建议

- **栈**：继续使用原生 HTML/CSS/JS 与现有 `static/` 结构即可；卡牌为独立视图时可单独 `card.html` + `card.js`/`card.css`，或嵌入现有单页通过「卡牌模式」开关切换 DOM 与事件绑定。  
- **组件化**：若后续卡面逻辑膨胀，可考虑轻量方案（如 Preact、Alpine）仅负责卡牌与战场视图，其余仍用现有 `app.js` 与 WebSocket。  
- **响应式**：竖版卡在窄屏可缩小宽度、两排战场改为上下堆叠，通过媒体查询与 flex 布局适配。
